## Актуальные несоответствия

### 1) Не реализован пользовательский доступ к разделу «Остатки» (после цикла)
- Требование: `4.8 Просмотр остатков после цикла`.
- Факт:
  - В навигации есть пункт `/remainders` (`frontend/src/widgets/layout/navigation/model/config.tsx`).
  - В роутере нет маршрута `/remainders` (`frontend/src/app/providers/RouterProvider/Router.tsx`).
  - Есть только UI-карточка `RemainderCard`, но нет страницы/фичи, которая получает и показывает список остатков.
- Риск: функционал из требований недоступен пользователю через SPA.

План исправления:
1. Добавить страницу `frontend/src/pages/remainders/RemaindersPage.tsx`.
2. Добавить feature для получения данных из `GET /api/remainders/get-user-remainders`.
3. Подключить маршрут `/remainders` в `Router.tsx` (через `ProtectedRoute`).
4. Отрисовать список через `entities/remainders/ui/RemainderCard` и состояние empty/error/loading.
5. Добавить smoke-проверку маршрута в README/frontend.

---

### 2) Нет реализации перераспределения остатков цикла в накопления/инвестиции
- Требование: `4.8 Перераспределение остатков: в накопления, в инвестиции`.
- Факт:
  - Остатки цикла сохраняются в таблицу `remainders` (scheduler: `backend/src/features/cycles/cycle.js`).
  - В API остатков есть только чтение: `backend/src/routes/remainders/get-user-remainders.js`.
  - Endpoint пополнения `POST /api/converts/replenish-convert` работает с `convert_type_limits`, а не с сущностью `remainders`.
- Риск: обязательная операция из требований отсутствует (есть только просмотр, без перераспределения именно цикловых остатков).

План исправления:
1. Добавить backend-роут `POST /api/remainders/redistribute`.
2. В транзакции:
   - валидировать остаток по `remainder_id`/`cycle_id` и владельца,
   - списывать сумму из `remainders`,
   - зачислять в целевой тип (`saving`/`investment`) через обновление конверта(ов) и/или лимитов.
3. Добавить frontend-форму на странице остатков (выбор суммы и целевого направления).
4. После мутации инвалидировать запросы: `remainders`, `converts`, `convert-limits`, `overview`.
5. Обновить README backend/frontend с DTO и примером вызова.

---

### 4) Не выполнен полный CRUD расходов в пользовательском интерфейсе
- Требование: `4.5 Расходы` + чек-лист `6. Есть ли CRUD для расходов`.
- Факт:
  - В backend CRUD есть (`add-expense`, `get-expenses`, `edit-expense`, `delete-expense`).
  - Во frontend реализованы просмотр и добавление; UI для редактирования/удаления расхода отсутствует (например `ExpenseListCard` не содержит действий).
- Риск: функциональное требование не закрыто на уровне приложения для конечного пользователя.

План исправления:
1. Добавить feature `features/expenses/delete-expense` (api + mutation + confirm modal).
2. Добавить feature `features/expenses/edit-expense` (форма редактирования, prefill, валидация).
3. Встроить действия в `entities/expense/ui/ExpenseListCard.tsx`.
4. Инвалидировать `get-expenses` и связанные overview-запросы после mutation.

---

### 5) Частичное несоответствие бизнес-правилу удаления конверта
- Требование: `5. При удалении конверта остаток возвращается на баланс`.
- Факт:
  - Сейчас удаление конверта запрещено, если есть расходы (`backend/src/routes/converts/delete-convert.js`, код `CONVERT_HAS_EXPENSES`).
  - Возврат остатка на «общий баланс» как отдельная операция/сущность явно не реализован.
- Риск: поведение отличается от формулировки требований (ожидается возврат остатка при удалении, а не блокировка сценария).

План исправления:
1. Согласовать целевую бизнес-модель «баланса» (user balance / type limit / pool).
2. Обновить `delete-convert` под согласованное правило:
   - вычислять остаток,
   - возвращать остаток в целевой баланс,
   - фиксировать операцию в транзакции.
3. Зафиксировать правило в README backend и в тестах.
