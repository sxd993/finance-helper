# Описание реализации приложения Finance Helper

## Общая характеристика приложения

Приложение **Finance Helper** представляет собой клиент-серверную информационную систему для ведения личного бюджета, распределения доходов по финансовым категориям, контроля расходов и анализа остатков денежных средств. Основная идея приложения заключается в том, что пользователь может не просто фиксировать расходы, а распределять свои средства по отдельным финансовым направлениям, которые в проекте называются **конвертами**. Такой подход позволяет заранее задавать лимиты на определённые группы трат, отслеживать доступный остаток по каждой категории и более осознанно подходить к управлению личными финансами.

С технической точки зрения приложение состоит из двух основных частей:

- клиентской части, реализованной на `React`, `TypeScript` и `Vite`;
- серверной части, реализованной на `Express` и `Sequelize`;
- базы данных, с которой сервер взаимодействует через ORM-модели;
- механизма авторизации на основе `JWT` и `httpOnly cookie`.

Архитектура проекта построена таким образом, чтобы разделить ответственность между уровнями системы. Клиент отвечает за визуализацию, взаимодействие с пользователем, хранение локального UI-состояния и обращение к API. Сервер отвечает за валидацию, бизнес-логику, работу с базой данных, проверку прав доступа и расчёт производных финансовых значений. Благодаря этому приложение остаётся достаточно структурированным и пригодным для расширения.

## Фирменный стиль и визуальная концепция интерфейса

Интерфейс приложения выполнен в современном минималистичном стиле. Основной акцент сделан на читаемости финансовой информации, простоте навигации и визуальном разделении сущностей. Поскольку приложение предназначено для ежедневного использования, в оформлении не применяются перегруженные декоративные элементы, а основное внимание уделено ясной структуре экранов, карточной подаче данных и цветовой дифференциации функциональных блоков.

В проекте используется светлая цветовая схема. Базовые цветовые переменные объявлены в глобальном файле стилей `frontend/src/App.css` через механизм `@theme`, предоставляемый Tailwind CSS v4. Это позволяет централизованно управлять ключевыми цветами интерфейса и переиспользовать их в различных компонентах.

Основными цветами интерфейса являются:

- `primary` `#2563eb` — основной акцентный цвет, используемый для значимых элементов интерфейса, иконок, выделения важных текстовых фрагментов и состояний фокуса;
- `primary-dark` `#1d4ed8` — затемнённый вариант основного цвета для состояний наведения и акцентных действий;
- `secondary` `#14b8a6` — дополнительный акцентный цвет, применяемый в кнопках действий, подтверждающих операциях и отдельных карточках;
- `secondary-dark` `#0e8376` — затемнённый вариант дополнительного цвета;
- `background` `#f9fafb` — базовый светлый фон приложения;
- `text` `#111827` — основной цвет текста.

Дополнительно в системе заданы отдельные цвета для доменных категорий, связанных с конвертами. Такое решение особенно важно в финансовом интерфейсе, так как позволяет пользователю быстро различать категории по цвету и не тратить время на повторное чтение заголовков.

Для типов конвертов используются следующие цветовые обозначения:

- важные траты — `#ef4444`;
- желания — `#facc15`;
- накопления — `#22c55e`;
- инвестиции — `#8b5cf6`;
- остатки — `#64748b`.

За счёт этого каждая финансовая категория приобретает собственную визуальную идентичность. Например, важные и обязательные расходы оформлены в более тёплом и тревожном красновато-оранжевом диапазоне, тогда как накопления и инвестиции ассоциируются с зелёным и фиолетовым, то есть цветами роста и долгосрочной перспективы.

В типографике интерфейса используется шрифт `Manrope`, указанный как основной в глобальных стилях приложения. Это современный гротескный шрифт, хорошо подходящий для цифровых интерфейсов за счёт высокой читаемости и спокойного характера начертания. Использование единого шрифта на всех экранах формирует целостное восприятие интерфейса и делает его визуально аккуратным.

Также в проекте присутствуют локальные шрифтовые ассеты `NotoSans Condensed`, однако в актуальной глобальной стилизации основным шрифтом остаётся именно `Manrope`. Это означает, что в текущей версии приложения главная типографическая роль закреплена за единым базовым шрифтом без усложнения визуальной системы.

Визуально интерфейс построен на нескольких устойчивых паттернах:

- использование карточек с белым фоном;
- скруглённые углы у контейнеров, кнопок и полей ввода;
- мягкие тени для отделения контента от общего фона;
- спокойные серые тона для вторичного текста;
- яркие акцентные цвета только в действительно важных местах.

Такой подход делает интерфейс визуально лёгким и не перегружает пользователя. Это особенно важно для финансового приложения, где приоритетом является концентрация на суммах, категориях, остатках и действиях, а не на декоративных элементах.

В качестве иконографики используется библиотека `lucide-react`. Иконки применяются в заголовках секций, нижней навигации, карточках и кнопках действий. Они помогают быстрее распознавать смысл разделов и служат дополнительным средством визуальной навигации. Например, для страницы расходов используется банковская иконка, для раздела остатков — монеты, для главной страницы — домашний экран, для настроек — стандартный символ конфигурации.

Отдельно стоит отметить адаптацию интерфейса под мобильный сценарий использования. Структура экранов ограничена по ширине контейнером `max-w-3xl`, а в нижней части интерфейса закреплена постоянная навигационная панель. Это делает приложение удобным для использования на смартфоне, где пользователь может быстро переключаться между основными разделами без возврата к стартовому экрану.

## Использование Tailwind CSS и UI-библиотек

Одной из ключевых особенностей клиентской части является использование `Tailwind CSS v4`. В проекте Tailwind подключён через `@import "tailwindcss"` в `App.css`, а стилизация компонентов выполняется непосредственно через utility-классы в JSX-разметке. Такой подход позволяет достаточно быстро собирать интерфейсы, не создавая большое количество отдельных CSS-файлов и не привязываясь к жёсткой BEM-структуре.

Преимущества такого решения в рамках данного проекта заключаются в следующем:

- стили находятся рядом с компонентом и легче читаются в контексте разметки;
- сокращается объём отдельного CSS-кода;
- проще поддерживать единый стиль через дизайн-токены;
- ускоряется разработка типовых UI-элементов;
- легче обеспечивать адаптивность интерфейса.

Примеры классов Tailwind используются практически во всех слоях фронтенда: от страниц и виджетов до мелких переиспользуемых компонентов. Через классы задаются отступы, размеры, цвета, тени, скругления, flex- и grid-раскладки, состояния `hover` и `focus`.

Кроме Tailwind CSS, в проекте подключён `Mantine`. На уровне приложения присутствует `AppMantineProvider`, который оборачивает всё дерево компонентов в `MantineProvider`. В текущей версии проекта Mantine используется скорее как инфраструктурная UI-зависимость и база для потенциального расширения интерфейса, тогда как основной визуальный слой в реальности построен именно на Tailwind-классах и собственных компонентах из `shared/ui`.

Таким образом, интерфейс реализован как комбинация:

- `Tailwind CSS` для основной стилизации;
- `Mantine` как подключённой UI-платформы;
- собственных компонентов общего назначения из слоя `shared/ui`;
- иконок `lucide-react` для навигации и смысловых акцентов.

## Архитектура клиентской части

Клиентская часть приложения реализована на `React 19` с использованием `TypeScript`. Сборка и локальная разработка выполняются с помощью `Vite`, что обеспечивает высокую скорость запуска проекта, быстрый hot reload и удобную работу в процессе разработки.

Важной особенностью фронтенда является применение архитектурного подхода **Feature-Sliced Design (FSD)**. Это позволяет разделить приложение по уровням ответственности и не смешивать доменные сущности, бизнес-логику, композиционные UI-блоки и страницы.

В проекте используются следующие слои:

- `app` — инфраструктурный уровень приложения, включающий провайдеры, роутинг, store и инициализацию;
- `entities` — базовые сущности предметной области и связанные с ними типы, компоненты и вспомогательная логика;
- `features` — пользовательские сценарии и прикладная логика;
- `widgets` — составные части интерфейса, объединяющие несколько feature- и entity-компонентов;
- `pages` — полноценные страницы приложения;
- `shared` — общие утилиты, UI-компоненты, API-клиент, хуки и вспомогательные функции.

Такое разбиение даёт ряд преимуществ. Во-первых, оно повышает читаемость кода: разработчику проще понять, где искать типы данных, где лежит логика работы с API, а где собираются готовые экранные блоки. Во-вторых, оно повышает переиспользуемость модулей. В-третьих, подобная структура делает проект более удобным для сопровождения и масштабирования.

### Уровень `app`

На уровне `app` сосредоточена базовая конфигурация приложения. Здесь находятся:

- подключение `QueryClientProvider` для TanStack Query;
- подключение `StoreProvider` для Redux Toolkit;
- подключение `MantineProvider`;
- конфигурация маршрутизации;
- инициализация корневого дерева приложения.

В файле `frontend/src/App.tsx` приложение оборачивается в несколько провайдеров:

- `QueryClientProvider` — для серверного состояния и кеширования запросов;
- `StoreProvider` — для Redux store;
- `AppMantineProvider` — для UI-провайдера Mantine;
- `RouterProvider` — для клиентской маршрутизации;
- `Toaster` из `sonner` — для показа всплывающих уведомлений.

В `frontend/src/main.tsx` приложение монтируется в DOM, а также регистрируется service worker. Это говорит о том, что проект частично ориентирован на PWA-сценарий и может использовать возможности браузерного кеширования и офлайн-поведения.

### Уровень `entities`

Слой `entities` содержит описания основных сущностей предметной области:

- `user`;
- `expense`;
- `convert`;
- `remainders`.

Для каждой сущности могут быть определены:

- TypeScript-типы;
- вспомогательные функции форматирования;
- доменные UI-компоненты;
- локальная логика отображения.

Например, для сущности `convert` определены типы данных, форматирование кода типа конверта, вычисление цвета по типу, карточки отдельных разновидностей конвертов и логика расчёта метрик карточек. Это показывает, что слой `entities` используется не только для “сухих” типов, но и для представления конкретных доменных объектов в интерфейсе.

### Уровень `features`

Слой `features` содержит прикладные сценарии, то есть действия, которые пользователь может выполнять в системе. Именно здесь сосредоточена основная бизнес-логика клиентской части.

В проекте реализованы, например, следующие feature-модули:

- авторизация пользователя;
- регистрация пользователя;
- выход из системы;
- получение типов конвертов;
- получение пользовательских конвертов;
- добавление конверта;
- редактирование конверта;
- удаление конверта;
- пополнение конверта;
- обновление инвестиционного конверта;
- добавление расхода;
- редактирование расхода;
- удаление расхода;
- фильтрация расходов;
- получение остатков;
- получение истории перераспределений;
- перераспределение остатков;
- изменение профиля;
- изменение процентного распределения бюджета;
- выбор иконки;
- переключение вкладок конвертов;
- получение количества оставшихся дней финансового цикла.

Каждая такая фича, как правило, включает:

- API-слой для вызова конкретного маршрута;
- хук или набор хуков для интеграции с React;
- форму или UI-компонент;
- локальные типы и вспомогательные функции.

Подобное построение делает код достаточно модульным: каждая функциональная возможность инкапсулирована и не размазывается по всему проекту.

### Уровень `widgets`

Слой `widgets` отвечает за более крупные составные блоки интерфейса. В отличие от `features`, здесь упор делается не на отдельное действие, а на сборку нескольких сущностей и фич в единый пользовательский блок.

К основным виджетам проекта можно отнести:

- `AuthForm` — контейнер авторизации и регистрации;
- `Header` — верхняя часть приложения;
- `Navigation` — нижняя навигационная панель;
- `ConvertOverview` — обзор текущего состояния по конвертам;
- `ConvertList` — список конвертов выбранного типа;
- `ExpensesOverview` — краткая сводка по расходам;
- `UserExpensesSection` — блок со списком расходов и фильтрами.

Виджеты являются важным слоем, потому что именно они формируют пользовательские экраны из более мелких кирпичиков и позволяют не перегружать страницы лишней логикой.

### Уровень `pages`

На уровне `pages` находятся полноценные маршрутизируемые экраны приложения. Это конечные точки пользовательского интерфейса, на которые попадает пользователь через роутер.

В проекте реализованы следующие основные страницы:

- `AuthPage` — страница входа и регистрации;
- `HomePage` — главная страница с вкладками конвертов, обзором и списком конвертов;
- `ExpensesPage` — страница со сводкой и списком расходов;
- `AddExpensesPage` — страница добавления нового расхода;
- `RemaindersPage` — страница просмотра остатков и истории перераспределения;
- `SettingsPage` — страница пользовательских настроек;
- `AddConvertsPage` — страница создания конверта;
- `ReplenishConvertPage` — страница пополнения конвертов;
- `NotFoundPage` — страница для неизвестных маршрутов.

Каждая страница строится как композиция из виджетов и фич, что соответствует идеологии FSD и делает экраны тонкими по логике.

## Маршрутизация на клиенте

Клиентская маршрутизация реализована через `react-router-dom`. Для приложения используется `createBrowserRouter`, а маршруты разделены на публичные и защищённые.

Публичным маршрутом является корневой путь:

- `/` — страница авторизации.

Защищённые маршруты доступны только авторизованному пользователю:

- `/home` — главная страница;
- `/converts/add-converts` — добавление нового конверта;
- `/converts/replenish` — пополнение конверта;
- `/expenses` — список расходов;
- `/expenses/add-expense` — добавление расхода;
- `/remainders` — просмотр остатков;
- `/settings` — настройки пользователя.

Если пользователь пытается открыть защищённый маршрут без авторизации, он перенаправляется на страницу входа. Если же авторизованный пользователь открывает публичный маршрут, система автоматически переводит его на `/home`.

Такая логика реализована через компоненты `ProtectedRoute` и `PublicRoute`. Оба компонента используют хук `useUser`, который обращается к серверному маршруту проверки пользователя. Пока выполняется запрос, пользователь видит состояние загрузки. После получения результата происходит либо отрисовка контента, либо перенаправление.

## Основные страницы и пользовательские сценарии

### Страница авторизации

Страница авторизации реализована через `AuthPage`, внутри которой используется виджет `AuthForm`. Этот виджет управляет локальным переключением между формой входа и формой регистрации через `useState`. Таким образом, пользователю не нужно переходить на отдельный экран для регистрации: оба сценария объединены в одном интерфейсном блоке.

Форма входа позволяет ввести логин и пароль, после чего вызывается мутация авторизации. При успехе происходит переход на главную страницу. Форма регистрации реализована сложнее: кроме базовых данных пользователя в проекте присутствует многошаговый onboarding-сценарий, для которого дополнительно используется Zustand-store.

### Главная страница

Главная страница `HomePage` является центральным экраном приложения. На ней пользователь работает с системой конвертов. Страница включает:

- переключатель вкладок типов конвертов;
- обзор текущего лимита и остатка;
- список конвертов выбранного типа;
- кнопку добавления нового конверта.

Таким образом, именно на главной странице сосредоточен основной сценарий распределения бюджета. Пользователь выбирает тип конверта, видит доступный лимит и список связанных категорий, после чего может создать новый конверт или перейти к работе с уже существующим.

### Страница расходов

Страница `ExpensesPage` предназначена для просмотра и управления расходами пользователя. В верхней части располагается обзорный блок, ниже — заголовок раздела и кнопка добавления расхода, а затем список расходов. Логика фильтрации вынесена в отдельную feature `expenses-filters`, что позволяет отбирать записи по типу конверта.

На этой странице пользователь может:

- просматривать список совершённых расходов;
- фильтровать расходы;
- переходить к созданию нового расхода;
- редактировать существующую запись;
- удалять расход.

### Страница остатков

Страница `RemaindersPage` показывает суммы, которые остались после завершения предыдущих финансовых циклов. Здесь присутствуют:

- карточка суммарного остатка;
- история перераспределения остатков;
- модальное окно для перевода остатка в выбранный конверт.

Эта часть приложения особенно полезна с точки зрения финансовой дисциплины, так как позволяет не терять неиспользованные средства, а осознанно перераспределять их между накопительными направлениями.

### Страница настроек

На странице `SettingsPage` собраны пользовательские параметры. В текущей реализации здесь доступны:

- изменение персональной информации;
- изменение процентного распределения дохода между финансовыми типами.

Это важный экран с точки зрения настройки поведения всей системы, поскольку проценты распределения влияют на лимиты и финансовую модель пользователя.

## Организация управления состоянием

В приложении используется не один механизм состояния, а комбинированный подход. Это связано с тем, что разные виды данных требуют разных инструментов.

### Redux Toolkit

Для хранения глобального клиентского состояния используется `Redux Toolkit`. В корневом reducer подключены следующие слайсы:

- `modal` — управление модальными окнами;
- `icon_picker` — выбранная иконка для расхода;
- `expenses_filters` — состояние фильтрации расходов;
- `convert_tabs` — активная вкладка типа конверта;
- `replenish_form` — состояние формы пополнения.

Использование Redux здесь оправдано тем, что это состояние требуется в нескольких местах интерфейса и не относится напрямую к серверным данным. Например, активная вкладка конверта используется одновременно в нескольких виджетах главной страницы, а состояние модальных окон должно быть доступно из разных компонентов.

### TanStack Query

Для работы с серверным состоянием используется `TanStack Query`. Это одно из ключевых решений проекта, поскольку приложение активно взаимодействует с API и должно:

- получать данные с сервера;
- кешировать их;
- избегать лишних повторных запросов;
- автоматически обновлять данные после изменений;
- удобно обрабатывать загрузку и ошибки.

В проекте через `useQuery` и `useMutation` реализованы:

- загрузка текущего пользователя;
- получение списка конвертов;
- получение типов конвертов;
- получение лимитов по типам;
- загрузка расходов;
- загрузка остатков;
- получение истории остатков;
- получение количества оставшихся дней цикла;
- создание, редактирование и удаление сущностей;
- обновление данных после мутаций через invalidation.

Для `QueryClient` в приложении заданы стандартные настройки, включая `staleTime`, что снижает число лишних запросов и делает интерфейс более отзывчивым.

### React Hook Form

Для форм в проекте используется `react-hook-form`. Это решение применено в форме авторизации, регистрации, создании расхода, редактировании расхода, создании конверта, изменении настроек и других сценариях, где требуется удобно работать с полями ввода, валидацией и отправкой формы.

Использование `react-hook-form` позволяет:

- уменьшить количество ручного кода;
- не хранить каждое поле в отдельном `useState`;
- удобнее валидировать данные;
- получать ошибки формы в структурированном виде;
- интегрировать формы с API-мутациями.

### Zustand

Хотя основным глобальным инструментом состояния является Redux Toolkit, в проекте также используется `Zustand`. Он подключён точечно для многошагового сценария регистрации и onboarding-потока. В частности, через Zustand хранится текущий шаг регистрации.

Такой выбор можно считать прагматичным: для изолированного сценария многошаговой формы нет необходимости перегружать общий Redux-store. Поэтому локально применён более лёгкий инструмент.

## Работа с API на клиенте

Взаимодействие клиента с сервером построено через единый экземпляр `Axios`, объявленный в `frontend/src/shared/api/client.tsx`. Этот клиент настроен следующим образом:

- `baseURL` установлен как `http://localhost:8000/api`;
- используется заголовок `Content-Type: application/json`;
- включён флаг `withCredentials: true`.

Последний пункт особенно важен, так как авторизация в приложении опирается на cookie. Благодаря `withCredentials` браузер автоматически передаёт cookie вместе с запросами к API.

Поверх общего клиента построены отдельные API-модули по фичам. Например:

- `AuthApi` — авторизация, регистрация, выход;
- API для расходов;
- API для конвертов;
- API для профиля;
- API для остатков;
- API для циклов.

Далее эти функции используются в пользовательских хуках и TanStack Query. Такой подход формирует понятную цепочку:

1. UI-компонент вызывает хук;
2. хук использует `useQuery` или `useMutation`;
3. query/mutation вызывает функцию API;
4. функция API обращается к серверу через единый `Axios`-клиент.

Эта схема делает приложение предсказуемым и удобным для сопровождения.

## Серверная часть приложения

Серверная часть реализована на `Express`. Это лёгкий и достаточно гибкий серверный фреймворк для Node.js, который хорошо подходит для построения REST API.

В файле `backend/src/app.js` выполняется основная конфигурация приложения:

- подключается `cors`;
- подключается `cookie-parser`;
- подключается `express.json()` для обработки JSON-запросов;
- настраивается список разрешённых источников;
- регистрируется `health`-маршрут;
- подключаются доменные роутеры.

Список разрешённых источников берётся из переменной окружения `CORS_ORIGINS`, а при её отсутствии используется адрес локального фронтенда `http://localhost:5173`. Это означает, что кросс-доменные запросы контролируются явно, а не открыты для любых клиентов.

В `backend/src/server.js` происходит:

- подключение переменных окружения;
- проверка соединения с базой данных;
- вывод списка endpoint-ов;
- запуск HTTP-сервера;
- запуск фонового планировщика циклов.

Последний пункт особенно интересен: в проекте есть отдельная feature для циклов, которая может выполнять периодические действия на сервере. Это показывает, что приложение не ограничивается только реакцией на пользовательские запросы, но также поддерживает фоновую бизнес-логику.

## Организация backend-маршрутов

Серверная часть разделена по доменам, и для каждого домена предусмотрен отдельный роутер. В `app.js` регистрируются следующие корневые группы маршрутов:

- `/api/auth`;
- `/api/converts`;
- `/api/expenses`;
- `/api/settings`;
- `/api/remainders`;
- `/api/cycles`.

Подобная структура соответствует принципу модульности: каждый блок отвечает за собственную область предметной логики.

### Маршруты авторизации

Блок `/api/auth` включает:

- `POST /api/auth/login` — вход пользователя в систему;
- `POST /api/auth/register` — регистрация нового пользователя;
- `GET /api/auth/check` — проверка текущей авторизации;
- `POST /api/auth/logout` — выход из системы.

При успешном входе и регистрации сервер возвращает данные пользователя и создаёт авторизационный токен. Проверка `check` используется клиентом для понимания, авторизован ли текущий пользователь и можно ли открыть защищённые маршруты.

### Маршруты конвертов

Блок `/api/converts` является одним из самых насыщенных по логике и включает:

- `GET /api/converts/get-convert-types` — получение справочника типов конвертов;
- `GET /api/converts/get-user-converts-type-limits` — получение лимитов по типам конвертов для пользователя;
- `POST /api/converts/add-convert` — создание нового конверта;
- `GET /api/converts/get-converts` — получение списка пользовательских конвертов;
- `GET /api/converts/converts-overview` — получение обзорной информации по конвертам;
- `PATCH /api/converts/:id/investment` — обновление данных инвестиционного конверта;
- `DELETE /api/converts/delete-convert/:id` — удаление конверта;
- `PATCH /api/converts/edit-convert/:id` — редактирование конверта;
- `POST /api/converts/replenish-convert` — пополнение конверта.

Именно через эти маршруты реализуется ключевая логика финансового распределения внутри приложения.

### Маршруты расходов

Блок `/api/expenses` включает:

- `GET /api/expenses/get-expenses` — получение расходов пользователя;
- `POST /api/expenses/add-expense` — добавление расхода;
- `PUT /api/expenses/edit-expense/:id` — редактирование расхода;
- `DELETE /api/expenses/delete-expense/:id` — удаление расхода.

Данный набор покрывает полный базовый CRUD-сценарий для сущности расхода.

### Маршруты настроек

Блок `/api/settings` включает:

- `PATCH /api/settings/change-percent` — изменение процентного распределения бюджета;
- `PATCH /api/settings/profile` — изменение профиля пользователя.

Через эти маршруты пользователь может корректировать базовые настройки, которые влияют на дальнейшую работу финансовой модели.

### Маршруты остатков

Блок `/api/remainders` включает:

- `GET /api/remainders/get-user-remainders` — получение остатка пользователя;
- `POST /api/remainders/redistribute` — перераспределение остатка;
- `GET /api/remainders/history` — история перераспределений.

Эти маршруты дают возможность не только видеть текущий доступный остаток, но и отслеживать прошлые действия с ним.

### Маршруты циклов

Блок `/api/cycles` включает:

- `GET /api/cycles/remaining-days` — получение количества оставшихся дней текущего финансового цикла.

Этот маршрут используется в интерфейсе для показа пользователю времени до сброса или пополнения финансовых лимитов.

## Работа с базой данных и Sequelize

Для работы с базой данных в проекте используется `Sequelize`. Модели располагаются в `backend/src/models`, а подключение и сборка моделей производятся через `backend/src/db/index.js`.

Среди основных моделей можно выделить:

- `User`;
- `Expense`;
- `Convert`;
- `ConvertType`;
- `ConvertTypeLimit`;
- `Cycle`;
- `Remainder`;
- `RemainderRedistribution`;
- `RemainderRedistributionItem`;
- специализированные модели для разновидностей конвертов, например накопительных и инвестиционных.

Наличие отдельных моделей для разных финансовых сущностей позволяет достаточно точно описать предметную область и не пытаться хранить всю бизнес-логику в одной универсальной таблице. Это особенно важно для конвертов, поскольку разные типы имеют разную финансовую природу: где-то важен лимит трат, где-то накопленная сумма, а где-то инвестиционная стоимость и текущая оценка.

Проект также следует хорошей практике: если операция затрагивает несколько сущностей и должна выполняться целостно, используется транзакция `sequelize.transaction`. Наиболее показательный пример — регистрация нового пользователя, где в рамках одной транзакции:

- создаётся пользователь;
- создаётся стартовый финансовый цикл;
- создаются лимиты по типам конвертов.

Такой подход обеспечивает согласованность данных и снижает риск появления частично сохранённых записей.

## Реализация авторизации и проверки токена

Авторизация в приложении реализована с использованием `JWT` и cookie-based механизма хранения токена. Это один из важнейших аспектов проекта, так как через него защищаются персональные финансовые данные пользователя.

В модуле `backend/src/utils/auth.js` сосредоточены основные утилиты для аутентификации:

- поиск пользователя по логину;
- преобразование пользователя к “публичному” формату без чувствительных данных;
- создание JWT-токена;
- проверка JWT-токена;
- хеширование пароля;
- сравнение пароля с хешем;
- запись авторизационной cookie;
- middleware `requireAuth`.

### Процесс регистрации

При регистрации выполняются следующие шаги:

1. Сервер принимает логин, имя, e-mail и пароль.
2. Проверяется, существует ли пользователь с таким логином.
3. Пароль хешируется через `bcryptjs`.
4. Создаётся пользователь.
5. Создаётся стартовый цикл.
6. Создаются стартовые лимиты по всем типам конвертов.
7. Генерируется JWT-токен.
8. Токен записывается в `httpOnly` cookie `token`.
9. Клиенту возвращаются данные пользователя и токен.

### Процесс входа

При входе алгоритм следующий:

1. Пользователь отправляет логин и пароль.
2. Сервер ищет пользователя по логину.
3. Через `bcrypt.compare` проверяется соответствие введённого пароля сохранённому хешу.
4. При успехе создаётся JWT-токен.
5. Токен сохраняется в cookie.
6. Клиент получает публичные данные пользователя.

### Проверка авторизации

Для проверки доступа к защищённым маршрутам используется middleware `requireAuth`. Логика её работы следующая:

1. Сервер пытается извлечь токен из заголовка `Authorization: Bearer ...`.
2. Если токен там отсутствует, проверяется наличие cookie `token`.
3. Если токен не найден, возвращается ошибка `401`.
4. Если токен найден, выполняется его верификация через `jwt.verify`.
5. Из токена извлекается идентификатор пользователя, в данном случае логин.
6. Сервер повторно ищет пользователя в базе.
7. Если пользователь найден, в `req.user` записываются публичные данные, а в `req.userId` — его идентификатор.
8. После этого запрос передаётся дальше в нужный обработчик.

Таким образом, все защищённые маршруты получают уже проверенного пользователя, а бизнес-логика может безопасно использовать его идентификатор для выборки данных только этого аккаунта.

### Особенности безопасности

Выбранная схема имеет несколько важных преимуществ:

- cookie помечается как `httpOnly`, поэтому JavaScript на клиенте не может напрямую читать токен;
- `sameSite: strict` снижает риск некоторых CSRF-сценариев;
- при production-режиме может использоваться флаг `secure`;
- пароль никогда не хранится в открытом виде, а только как хеш;
- сервер не доверяет клиенту напрямую и всегда повторно ищет пользователя в базе после проверки токена.

Для учебного проекта это уже достаточно корректная и взрослая схема авторизации.

## Реализация основных пользовательских сценариев

### Создание и ведение конвертов

Одна из центральных функций приложения — работа с конвертами. Пользователь может создавать конверты разных типов, хранить в них лимиты, текущие суммы или инвестиционные значения и дальше использовать их как основу для финансового планирования.

На клиенте для этого предусмотрены:

- формы создания конвертов;
- карточки отображения конвертов;
- модальные окна редактирования и удаления;
- отдельный сценарий пополнения конверта;
- обзор лимитов и доступных сумм.

Разделение по типам позволяет учитывать специфику разных финансовых целей. Например, накопительный конверт логически отличается от конверта текущих трат или инвестиционного направления, поэтому и форма, и серверная обработка различаются.

### Учёт расходов

Второй важнейший сценарий — учёт расходов. Пользователь может добавлять расход, привязывать его к конкретному конверту и фиксировать сумму. В перспективе это позволяет контролировать, насколько фактические траты укладываются в распределённый бюджет.

На клиенте для этого предусмотрены:

- форма добавления расхода;
- выбор типа конверта;
- выбор конкретного конверта;
- выбор иконки расхода;
- список расходов;
- редактирование и удаление.

Использование выбора иконки делает список расходов визуально более понятным и удобным для восприятия.

### Работа с остатками

Дополнительный финансовый сценарий — работа с остатками. Если по завершении цикла часть средств остаётся неиспользованной, приложение не “теряет” её, а позволяет вывести эту сумму в отдельный раздел и распределить дальше. Это делает систему более приближённой к реальной практике финансового планирования.

На уровне интерфейса пользователь видит:

- общий доступный остаток;
- историю всех перераспределений;
- форму перевода остатка в другой финансовый тип.

Таким образом, остатки становятся не побочным эффектом расчётов, а полноценной частью пользовательской модели.

## Анализ используемых и потенциально лишних маршрутов

В процессе анализа проекта можно заметить, что большая часть серверных маршрутов реально используется клиентом. Однако есть и маршруты, которые в текущем пользовательском интерфейсе либо задействованы ограниченно, либо вообще не встроены в основной экранный поток.

Например, маршрут:

- `GET /api/converts/converts-overview`

существует на сервере и имеет отдельную реализацию, однако в текущей клиентской логике основной обзор на главной странице строится на запросе лимитов `get-user-converts-type-limits`, а не на прямом использовании этого endpoint-а в рабочем UI-сценарии. В кодовой базе клиентская функция для него есть, но фактически маршрут выглядит кандидатом на ревизию.

Также внимания заслуживает маршрут:

- `PATCH /api/converts/:id/investment`

Для него присутствуют API-обёртка, хук и форма, однако в основном пользовательском потоке этот сценарий интегрирован не так явно, как, например, создание, удаление или редактирование конвертов. Это тоже делает маршрут потенциальным кандидатом на упрощение или доработку.

С практической точки зрения это означает, что проект уже содержит основу для рефакторинга: можно дополнительно проверить фактическую востребованность отдельных endpoint-ов и убрать устаревшие или полузавершённые части API. Для курсовой работы это даже полезно отметить, поскольку показывает понимание не только реализованных возможностей, но и перспектив сопровождения проекта.

## Преимущества выбранных архитектурных решений

По результатам анализа можно выделить несколько сильных сторон реализации.

Во-первых, приложение имеет чёткое разделение на клиент и сервер. Это делает систему масштабируемой и позволяет независимо развивать пользовательский интерфейс и backend-логику.

Во-вторых, на клиенте применяется Feature-Sliced Design, что положительно влияет на читаемость и поддержку кода. Код не свален в набор случайных компонентов, а разделён по назначению.

В-третьих, грамотно разделены виды состояния:

- Redux Toolkit используется для глобального UI-состояния;
- TanStack Query — для серверных данных;
- React Hook Form — для форм;
- Zustand — для изолированного многошагового сценария.

Такое распределение показывает, что в проекте не выбран один инструмент “на всё”, а каждая задача решается подходящим средством.

В-четвёртых, авторизация реализована достаточно корректно даже по меркам прикладной промышленной разработки: используются хеширование паролей, JWT, cookie, middleware проверки и повторная валидация пользователя через базу данных.

В-пятых, интерфейс построен на современном адаптивном стеке с Tailwind CSS и продуманной визуальной системой, где категории различаются не только текстом, но и цветом, иконками и структурой карточек.

## Заключение

В ходе анализа реализации приложения **Finance Helper** можно сделать вывод, что оно представляет собой цельную и хорошо организованную клиент-серверную систему для управления личными финансами. Проект охватывает основные задачи пользователя: регистрацию и авторизацию, настройку финансовой модели, создание конвертов, учёт расходов, отслеживание остатков и управление параметрами бюджета.

Клиентская часть приложения построена на современном технологическом стеке: `React`, `TypeScript`, `Vite`, `Tailwind CSS`, `Redux Toolkit`, `TanStack Query`, `React Hook Form`. Использование Feature-Sliced Design позволило логично структурировать код и развести инфраструктуру, сущности, бизнес-логику, виджеты и страницы. Благодаря этому проект легко читать, поддерживать и расширять.

Серверная часть на `Express` и `Sequelize` обеспечивает обработку доменных сценариев, работу с базой данных, проверку авторизации и выполнение фоновой логики. Разделение backend-маршрутов по доменам делает API понятным и организованным. Использование транзакций в критически важных местах дополнительно повышает надёжность хранения данных.

Отдельного внимания заслуживает реализация авторизации. Использование `JWT`, `bcryptjs`, `httpOnly cookie` и middleware `requireAuth` позволяет надёжно защищать пользовательские данные и ограничивать доступ к приватным маршрутам.

С точки зрения интерфейса приложение оформлено в едином фирменном стиле: используется светлая современная палитра, единый шрифт, карточная структура, цветовая дифференциация типов конвертов и удобная мобильная навигация. Всё это делает систему не только технически работоспособной, но и удобной для конечного пользователя.

Итоговый вывод состоит в том, что приложение реализовано на актуальных технологиях, имеет продуманную архитектуру и демонстрирует комплексный подход к разработке современной веб-системы. Для курсовой работы такой проект является удачным примером практического применения архитектурных паттернов, клиент-серверного взаимодействия, управления состоянием, авторизации и построения пользовательского интерфейса.
