# Описание таблиц базы данных

Этот файл нужен как краткое пояснение смысла БД проекта `finance-helper`: какие таблицы есть, зачем они нужны и как участвуют в логике приложения.

Источник структуры:
- SQL-схема: `backend/db/schema_v2.sql`
- Модели Sequelize: `backend/src/models`

## Общая идея БД

База данных поддерживает систему финансовых "конвертов":
- пользователь задает доход и проценты распределения;
- система рассчитывает лимиты по типам конвертов;
- пользователь создает конкретные конверты;
- по конвертам ведется история пополнений и трат;
- в конце цикла сохраняются остатки, которые потом можно перераспределять.

Архитектурно БД делится на 5 смысловых блоков:
- пользователи и их настройки;
- справочники и типы конвертов;
- сами конверты и их специализированные данные;
- журнал операций;
- циклы, остатки и перераспределение остатков.

## 1. `users`

### Назначение
Главная таблица пользователей. Хранит не только учетные данные, но и финансовые настройки, от которых зависит работа всего приложения.

### Что хранится
- логин, имя, email;
- хэш пароля;
- месячный доход;
- тип цикла;
- проценты распределения дохода по категориям:
  - `percent_important`
  - `percent_wishes`
  - `percent_saving`
  - `percent_investment`
- даты создания и обновления.

### Бизнес-смысл
Это "корневая" сущность всей БД. Почти все остальные таблицы привязаны к пользователю. На основании дохода и процентов пользователя создаются лимиты по типам конвертов и стартовый финансовый цикл.

### Связи
- один пользователь имеет много `cycles`;
- один пользователь имеет много `converts`;
- один пользователь имеет много `operations`;
- один пользователь имеет много `convert_type_limits`;
- один пользователь имеет много `remainders`;
- один пользователь имеет много `remainder_redistributions`.

## 2. `convert_types`

### Назначение
Справочник типов конвертов. Это не пользовательские данные, а фиксированный набор категорий системы.

### Что хранится
- код типа: `important`, `wishes`, `saving`, `investment`;
- человекочитаемое название;
- описание;
- служебные признаки:
  - сбрасывается ли тип при новом цикле;
  - есть ли у него лимит;
  - допускает ли он траты;
- порядок отображения.

### Бизнес-смысл
Таблица описывает правила поведения каждого типа конвертов. Например:
- `important` и `wishes` поддерживают траты;
- `saving` и `investment` предназначены для накопления;
- часть типов участвует в сбросе остатков при закрытии цикла.

### Связи
- один тип связан со многими `converts`;
- один тип используется во многих `operations`;
- один тип используется во многих `convert_type_limits`;
- один тип используется во многих `remainders`;
- один тип может быть целевым типом во многих `remainder_redistributions`.

## 3. `cycles`

### Назначение
Финансовые циклы пользователя. Цикл нужен для периодического учета лимитов, расходов и остатков.

### Что хранится
- пользователь;
- дата начала;
- дата окончания;
- признак закрытия;
- время закрытия;
- дата создания записи.

### Бизнес-смысл
Цикл отражает временной период учета бюджета. Когда цикл закрывается:
- система фиксирует остатки;
- открывает новый цикл;
- пересчитывает лимиты;
- начинает новый период учета.

### Связи
- каждый цикл принадлежит одному пользователю;
- один цикл может иметь несколько записей в `remainders`.

## 4. `converts`

### Назначение
Базовая таблица конвертов пользователя. Здесь хранится общая информация о каждом конверте независимо от его типа.

### Что хранится
- пользователь-владелец;
- тип конверта;
- название конверта;
- признак активности;
- даты создания и обновления.

### Бизнес-смысл
Это центральная таблица предметной области. Конверт — это конкретная финансовая сущность пользователя, например:
- "Продукты" в типе `important`;
- "Отпуск" в типе `saving`;
- "ETF" в типе `investment`.

Базовая таблица хранит общее, а типоспецифичные поля вынесены в отдельные подтаблицы.

### Связи
- каждый конверт принадлежит одному пользователю;
- каждый конверт имеет один `convert_type`;
- у конверта может быть одна специализированная запись:
  - в `convert_spend`
  - или в `convert_saving`
  - или в `convert_investment`
- один конверт может иметь много `operations`.

## 5. `convert_spend`

### Назначение
Подтаблица для расходных конвертов. Используется для типов, из которых можно тратить деньги.

### Что хранится
- ссылка на конверт;
- месячный лимит;
- профинансированная сумма.

### Бизнес-смысл
Применяется для конвертов типов `important` и `wishes`. Таблица нужна, чтобы отделить расходные конверты от накопительных и инвестиционных.

`funded_amount` показывает, сколько денег было выделено конверту в текущей логике бюджета, а фактический остаток определяется с учетом расходов из `operations`.

### Связи
- одна запись `convert_spend` соответствует одному `converts`.

## 6. `convert_saving`

### Назначение
Подтаблица для накопительных конвертов.

### Что хранится
- ссылка на конверт;
- целевая сумма;
- уже накопленная сумма.

### Бизнес-смысл
Используется для конвертов типа `saving`. Позволяет хранить прогресс накопления отдельно от расходной логики.

### Связи
- одна запись `convert_saving` соответствует одному `converts`.

## 7. `convert_investment`

### Назначение
Подтаблица для инвестиционных конвертов.

### Что хранится
- ссылка на конверт;
- сумма вложений;
- текущая оценка стоимости.

### Бизнес-смысл
Используется для типа `investment`. В отличие от накоплений, здесь есть разделение между вложенной базой и текущей стоимостью, чтобы можно было считать рост или просадку актива.

### Связи
- одна запись `convert_investment` соответствует одному `converts`.

## 8. `convert_type_limits`

### Назначение
Лимиты бюджета по типам конвертов на пользователя.

### Что хранится
- пользователя;
- тип конверта;
- сумму лимита;
- дату обновления.

### Бизнес-смысл
Это не лимит отдельного конверта, а лимит всей категории. Например:
- сколько всего денег пользователь может выделить на `important`;
- сколько на `wishes`;
- сколько на `saving`.

Лимиты рассчитываются на основе дохода и процентного распределения пользователя.

### Связи
- каждая запись принадлежит одному пользователю;
- каждая запись привязана к одному типу из `convert_types`.

### Особенность
Первичный ключ составной: `(user_id, type_code)`. Это означает, что у пользователя может быть только один лимит на каждый тип.

## 9. `operations`

### Назначение
Единый журнал всех финансовых операций в системе.

### Что хранится
- пользователя;
- тип операции:
  - `expense`
  - `replenishment`
- источник операции:
  - `spend`
  - `type_limit`
  - `remainder`
- сумму;
- время операции;
- ссылку на конверт;
- снимок данных конверта на момент операции:
  - `convert_name`
  - `convert_type`
- название операции;
- иконку;
- ссылку на перераспределение остатка, если операция возникла из этой логики.

### Бизнес-смысл
Это одна из самых важных таблиц БД. Она хранит полную историю движения денег.

Через нее реализованы:
- траты;
- пополнения конвертов;
- история действий пользователя;
- отображение финансовой хроники на фронтенде.

### Важная особенность
Отдельной таблицы `expenses` нет. Расходы — это строки в `operations`, где:
- `type = 'expense'`.

Пополнения тоже записываются сюда, но с:
- `type = 'replenishment'`.

### Почему это удобно
Такой подход дает единый журнал событий и упрощает:
- построение истории;
- фильтрацию операций;
- аналитику по движению денег.

### Связи
- каждая операция принадлежит одному пользователю;
- операция может быть привязана к одному конверту;
- операция связана с типом конверта;
- операция может быть связана с одной записью `remainder_redistributions`.

## 10. `remainders`

### Назначение
Остатки средств после закрытия финансового цикла.

### Что хранится
- пользователя;
- цикл;
- тип конверта;
- сумму остатка;
- дату создания записи.

### Бизнес-смысл
Когда цикл закрывается, система считает, какие деньги остались в сбрасываемых категориях, и сохраняет их в этой таблице.

Это уже не остаток конкретного пользовательского конверта, а агрегированный остаток по типу за конкретный цикл.

### Пример
Если по типу `important` к концу цикла осталось 3 000, то появляется одна запись в `remainders`, которая фиксирует этот остаток для данного цикла.

### Связи
- каждая запись принадлежит одному пользователю;
- каждая запись относится к одному циклу;
- каждая запись связана с одним типом конверта.

### Ограничение
На один цикл и один тип допускается только одна запись остатка.

## 11. `remainder_redistributions`

### Назначение
Заголовок операции перераспределения остатков.

### Что хранится
- пользователя;
- целевой конверт;
- целевой тип;
- общую сумму перевода;
- дату создания.

### Бизнес-смысл
Если пользователь переводит накопленные остатки в конверт накоплений или инвестиций, сама операция перевода фиксируется здесь.

Это "шапка" события перераспределения:
- куда перевели;
- сколько перевели;
- кто перевел.

### Связи
- принадлежит одному пользователю;
- указывает на один целевой конверт;
- указывает на один целевой тип;
- имеет много строк детализации в `remainder_redistribution_items`.

## 12. `remainder_redistribution_items`

### Назначение
Детализация перераспределения остатков по источникам.

### Что хранится
- ссылку на перераспределение;
- ссылку на конкретный остаток;
- сумму, взятую именно из этого остатка.

### Бизнес-смысл
Если одно перераспределение собирается из нескольких остатков, эта таблица показывает, из каких именно записей `remainders` была набрана итоговая сумма.

Иными словами:
- `remainder_redistributions` отвечает на вопрос "что сделали в целом?";
- `remainder_redistribution_items` отвечает на вопрос "из чего именно это было собрано?".

### Связи
- каждая строка принадлежит одному `remainder_redistributions`;
- каждая строка ссылается на один `remainders`.

## Как объяснить смысл БД в одном абзаце

Эта БД обслуживает приложение для личного финансового планирования по системе конвертов. Пользователь задает доход и проценты распределения, система создает лимиты по категориям, пользователь заводит конверты, пополняет их и тратит деньги, а все движения записываются в единый журнал `operations`. Финансовый учет идет по циклам, после завершения цикла остатки фиксируются в `remainders` и при необходимости перераспределяются в накопления или инвестиции через связку `remainder_redistributions` и `remainder_redistribution_items`.

## Быстрая карта таблиц

| Таблица | Роль |
|---|---|
| `users` | Пользователь и его финансовые настройки |
| `convert_types` | Справочник типов конвертов |
| `cycles` | Финансовые циклы пользователя |
| `converts` | Базовые пользовательские конверты |
| `convert_spend` | Данные расходных конвертов |
| `convert_saving` | Данные накопительных конвертов |
| `convert_investment` | Данные инвестиционных конвертов |
| `convert_type_limits` | Лимиты по категориям конвертов |
| `operations` | Единый журнал трат и пополнений |
| `remainders` | Остатки после закрытия цикла |
| `remainder_redistributions` | Заголовки переводов остатков |
| `remainder_redistribution_items` | Детализация, из каких остатков собран перевод |
