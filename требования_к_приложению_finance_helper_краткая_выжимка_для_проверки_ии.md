# 📌 КРАТКАЯ ВЫЖИМКА ТРЕБОВАНИЙ (для автоматической проверки)

## 1. Тип системы
- Веб-приложение (SPA)
- Назначение: управление личными финансами
- Архитектура: клиент-серверная

---

## 2. Технологии (обязательные)

### Frontend
- React
- SPA (без перезагрузки страницы)
- Архитектура: Feature-Sliced Design (FSD)

### Backend
- Node.js
- Express
- REST API (HTTP + JSON)

### Database
- Реляционная БД: MySQL
- ORM: Sequelize

---

## 3. Архитектурные требования
- Чёткое разделение: frontend / backend / database
- Взаимодействие через REST API
- Модульная структура backend
- Использование middleware для авторизации
- Поддержка транзакций в критичных операциях

---

## 4. Основной функционал (обязательно)

### 4.1 Авторизация
- Регистрация пользователя
- Авторизация пользователя

### 4.2 Профиль
- Просмотр профиля
- Редактирование данных
- Настройка процентного распределения бюджета

### 4.3 Доход и распределение
- Ввод дохода
- Автоматическое распределение по категориям:
  - необходимые расходы
  - желания
  - накопления
  - инвестиции

### 4.4 Конверты (бюджеты)
- Создание конвертов
- Редактирование конвертов
- Удаление конвертов
- Просмотр списка конвертов

### 4.5 Расходы
- Добавление расхода
- Выбор конверта
- Проверка достаточности средств
- Списание средств при успехе
- Ошибка при недостатке средств
- Удаление расхода (с возвратом суммы)
- Просмотр списка расходов

### 4.6 Балансы и аналитика
- Обновление балансов после операций
- Отображение текущих остатков

### 4.7 Финансовый цикл
- Учёт месячного цикла
- Отображение дней до конца цикла
- Хранение циклов

### 4.8 Остатки средств
- Просмотр остатков после цикла
- Перераспределение остатков:
  - в накопления
  - в инвестиции
- (опционально) автоматическое перераспределение через планировщик

---

## 5. Бизнес-логика (строго проверить)
- Нельзя списать больше средств, чем есть в конверте
- После каждой операции обновляются балансы
- При удалении расхода сумма возвращается
- При удалении конверта остаток возвращается на баланс
- Распределение дохода происходит автоматически

---

## 6. Структура базы данных (ключевые сущности)
- users
- convert_types
- converts
- expenses
- cycles
- convert_type_limits
- remainders

Связи:
- пользователь → конверты / расходы / циклы

---

## 7. Нефункциональные требования
- Быстрый отклик интерфейса (SPA)
- Актуальность данных без перезагрузки
- Масштабируемая архитектура
- Модульность кода
- Безопасность (аутентификация, защита данных)
- Кроссплатформенность (desktop / mobile / tablet)

---

## 8. Проверочный чек-лист (для ИИ агента)

Проверь:

1. Есть ли разделение frontend/backend/database
2. Используется ли React + Node.js + MySQL
3. Есть ли REST API
4. Реализована ли авторизация
5. Есть ли CRUD для конвертов
6. Есть ли CRUD для расходов
7. Есть ли проверка баланса перед списанием
8. Обновляются ли балансы после операций
9. Есть ли автоматическое распределение дохода
10. Есть ли финансовые циклы
11. Есть ли работа с остатками
12. Используется ли ORM
13. Есть ли защита маршрутов (middleware)
14. SPA ли это приложение

---

## 9. Критерий соответствия

Приложение считается соответствующим, если:
- Выполнены ВСЕ функциональные требования
- Соблюдена архитектура
- Используются указанные технологии
- Реализована ключевая бизнес-логика

Любое отклонение = несоответствие требованиям
